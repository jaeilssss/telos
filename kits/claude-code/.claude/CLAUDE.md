# 명세 우선(Spec-First) 개발 규칙 (전역)

이 규칙은 사용자 레벨(`~/.claude/CLAUDE.md`)에 있어 **모든 프로젝트**에 적용된다.
핵심: *AI는 무엇이든 만들 수 있다. 어려운 건 무엇을 만들지 아는 것이다.*

## 절대 규칙

1. **스펙 없이 구현 금지.** 새 기능/모듈 구현 전, 해당 **프로젝트 루트**에 `SPEC.md`가
   있고 목표·제약·인수 기준(acceptance criteria)이 구체적이어야 한다. 없거나 모호하면
   코드를 쓰지 말고 `/spec` 을 먼저 돌리도록 사용자에게 요청한다.
2. **항상 SPEC.md를 먼저 읽는다.** 구현·수정·평가 시작 시 프로젝트의 `SPEC.md` 를 읽고
   그 인수 기준을 기준점으로 삼는다.
3. **추측하지 말고 질문한다.** 모호함을 임의의 가정으로 메우지 않는다.

> 참고: 스펙 게이트 훅은 **차단이 아니라 경고**다. 일회성 스크립트나 탐색성 작업처럼
> 스펙이 과한 경우엔 무시하고 진행해도 된다. 규칙은 "제품 기능"에 적용한다는 의도다.

## 워크플로우 (우로보로스 4단계의 경량 버전)

```
/spec   →  소크라테스식 질문으로 모호함 제거 → 프로젝트 루트에 SPEC.md(계약) 생성
구현    →  SPEC.md 기준으로 구현
/eval   →  3단계 게이트 (Mechanical → Semantic → (선택) Consensus)
반복    →  실패한 인수 기준을 SPEC.md 에 되먹이고 루프
```

`/spec` 은 열린 질문이 모두 소진되면 `spec-ambiguity-evaluator` 서브에이전트로
모호성 점검을 자동 수행하고, Ambiguity <= 0.2 일 때만 `frozen` 으로 전환한다.

## 3단계 평가 게이트 (비용 순서)

- **Stage 1 — Mechanical ($0):** 테스트·린트·타입체크·빌드. 통과 못 하면 여기서 멈춘다.
  LLM 토큰을 쓰기 전에 기계가 먼저 거른다.
- **Stage 2 — Semantic:** `spec-evaluator` 서브에이전트로 각 인수 기준 충족 여부를 근거와 함께 판정.
- **Stage 3 — Consensus (선택):** 고위험이거나 Stage 2가 불확실할 때만 다른 모델로 교차 검증.
